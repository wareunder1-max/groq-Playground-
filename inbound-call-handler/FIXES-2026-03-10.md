## Latest Fix (Final Solution)

### Recording Fix v3 - Direct HTTP API (WORKING)

**Problem:**
- ari-client library's `channel.record()` method returns "Internal Server Error"
- Manual Postman API calls work perfectly with same parameters
- Issue is with the ari-client library wrapper, not Asterisk itself

**Solution:**
- Bypass ari-client library's `channel.record()` wrapper
- Use direct HTTP POST to ARI endpoint with axios (like AVA project does)
- Use query parameters instead of request body
- Listen for recording events via ARI WebSocket

**Code Changes:**
```javascript
// NEW: Direct HTTP POST to ARI endpoint
const recordUrl = `http://${process.env.ASTERISK_HOST}:${process.env.ASTERISK_PORT}/ari/channels/${channel.id}/record`;

const recordParams = {
  name: recordingName,
  format: 'wav',
  maxDurationSeconds: 30,
  maxSilenceSeconds: 3,
  ifExists: 'overwrite',
  beep: false,
  terminateOn: 'none'
};

const recordResponse = await axios.post(recordUrl, null, {
  params: recordParams,  // Query parameters, not body
  auth: {
    username: process.env.ASTERISK_USERNAME,
    password: process.env.ASTERISK_PASSWORD
  }
});

// Listen for recording events via ARI WebSocket
ari.on('RecordingFinished', finishedHandler);
ari.on('RecordingFailed', failedHandler);
```

**Why This Works:**
1. **Direct API access**: Bypasses buggy ari-client wrapper
2. **Query parameters**: ARI expects params in query string, not body
3. **Proven approach**: AVA project uses same method successfully
4. **Event handling**: Still uses ARI WebSocket for recording events

## Changes Made

### 1. Recording Fix (server.js)

**Before:**
- Used direct HTTP POST to ARI endpoint
- Bypassed proper event handling
- Used arbitrary 32-second timeout

**After:**
- Use ari-client library's `channel.record()` method
- Properly handle `RecordingFinished` and `RecordingFailed` events
- Use simple filename without path (e.g., `ai_recording_1234567890`)
- Asterisk automatically saves to `/var/spool/asterisk/monitor/`
- Wait for actual recording events instead of timeout

**Code Changes:**
```javascript
// NEW: Use ari-client library with proper event handling
const recording = ari.LiveRecording();

await channel.record({
  name: recordingName,  // Simple filename, no path
  format: 'wav',
  ifExists: 'overwrite',
  maxDurationSeconds: 30,
  maxSilenceSeconds: 3,
  beep: false,
  terminateOn: 'none'
}, recording);

// Wait for recording events
await new Promise((resolve, reject) => {
  recording.on('RecordingFinished', () => resolve());
  recording.on('RecordingFailed', (event) => reject(new Error('Recording failed')));
});
```

### 2. TTS Playback Fix (server.js)

**Before:**
- Saved files to `/var/lib/asterisk/sounds/custom/` (read-only)
- Used relative path: `sound:custom/filename`

**After:**
- Save files to `/tmp/asterisk-audio/` (writable)
- Use absolute path: `sound:/tmp/asterisk-audio/filename`
- Remove .mp3 extension (Asterisk adds it automatically)

**Code Changes:**
```javascript
// NEW: Use /tmp for TTS files
const AUDIO_DIR = '/tmp/asterisk-audio';

// NEW: Use absolute path for playback
const filePath = path.join(AUDIO_DIR, fileName);
const filePathWithoutExt = filePath.replace('.mp3', '');
await channel.play({ media: `sound:${filePathWithoutExt}` }, playback);
```

## Why These Fixes Work

### Recording Fix
1. **Event-driven approach**: The ari-client library properly handles ARI WebSocket events
2. **Correct path handling**: Asterisk expects just a filename, not a full path
3. **Proper waiting**: Wait for actual recording completion events instead of arbitrary timeouts

### TTS Playback Fix
1. **Writable directory**: `/tmp/` is always writable, unlike `/var/lib/asterisk/sounds/`
2. **Absolute paths**: Asterisk can find files using absolute paths
3. **Extension handling**: Asterisk automatically adds the extension when searching for files
